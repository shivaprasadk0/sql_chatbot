# Legal & Compliance Assessment

**Project:** AI Knowledge Assistant (SQL Query Chatbot)
**Date:** 2026-02-20
**Risk Level:** HIGH (executes AI-generated code on live databases, potential PII access)

---

## Executive Summary

The application generates and executes SQL queries on databases using AI. This creates unique legal risks: liability for incorrect queries, data privacy obligations, AI-specific regulations, and intellectual property considerations. Currently, no legal safeguards exist in the codebase — no license, no terms of service, no data privacy mechanisms, and no disclaimers. This document identifies all legal and compliance gaps.

---

## 1. Software License

### Current State: NO LICENSE

The repository has no LICENSE file. Under default copyright law, this means:
- **No one has permission** to use, modify, or distribute the code
- **All rights are reserved** by the author
- If this is a startup with multiple contributors, there is no legal clarity on code ownership

### Required Action

Choose and add a license:

| License | Use Case | Key Terms |
|---------|---------|-----------|
| MIT | Open-source, maximum freedom | Anyone can use/modify/distribute |
| Apache 2.0 | Open-source with patent protection | Like MIT but includes patent grant |
| Proprietary | Commercial product | All rights reserved, no redistribution |
| BSL (Business Source License) | Source-available, delayed open-source | Commercial use requires license, becomes open-source after X years |

**Recommendation for a startup building a commercial product:**
- If keeping code private: **Proprietary** (no LICENSE file needed in private repo, but should have internal IP agreement)
- If open-sourcing: **Apache 2.0** (patent protection is important for AI products)
- If source-available: **BSL** (used by companies like MariaDB, Sentry, HashiCorp)

### Additional IP Requirements

1. **Contributor License Agreement (CLA)** — If accepting external contributions, require contributors to sign a CLA
2. **Employee IP Assignment** — Ensure all employees/contractors have signed IP assignment agreements
3. **Invention disclosure** — If any novel algorithms are created, consider patent filing

## 3. AI-Generated Output Liability

### The Core Legal Risk

This application generates SQL queries using AI and executes them on a live database. This creates liability risk in several scenarios:

#### Scenario 1: Incorrect Query Results
- User asks: "What were our total sales last quarter?"
- AI generates SQL that calculates incorrectly (wrong date range, wrong aggregation)
- User makes business decisions based on wrong data
- **Risk:** Liability for incorrect business intelligence

#### Scenario 2: Destructive Query
- Despite safety measures, AI generates a query that modifies data
- Data is corrupted or deleted
- **Risk:** Liability for data loss

#### Scenario 3: Data Breach via Query
- User crafts a prompt that tricks the AI into querying sensitive data
- User gains access to PII they shouldn't see
- **Risk:** Data breach liability, regulatory violations

### Required Mitigations

1. **Disclaimers in Terms of Service:**
   ```
   The SQL queries generated by this service are AI-generated and may contain 
   errors. Users should verify all query results before making business decisions. 
   [Company] is not liable for decisions made based on AI-generated query results.
   ```

2. **In-application disclaimers:**
   - Display "AI-generated query — verify results before use" on every response
   - Show the SQL query to the user so they can review it before execution
   - Optional: Require user confirmation before executing the query

3. **Technical safeguards:**
   - Read-only database access (prevents data modification)
   - Query validation (prevents dangerous queries)
   - Audit logging (evidence of what was queried and by whom)
   - Table/column access controls (prevent access to sensitive data)

4. **Insurance:**
   - Consider Errors & Omissions (E&O) insurance
   - Consider Cyber Liability insurance

## 4. Data Privacy & Protection

### Applicable Regulations

| Regulation | Applies If | Key Requirements |
|------------|-----------|-----------------|
| GDPR (EU) | Any EU users or EU data subjects | Consent, right to deletion, data minimization, DPA with processors |
| CCPA/CPRA (California) | California residents' data | Disclosure, opt-out of sale, deletion rights |
| SOC 2 | Enterprise customers require it | Security controls, audit trails |
| HIPAA (US Healthcare) | Healthcare data | Encryption, access controls, BAA with vendors |
| PCI DSS | Payment card data | Encryption, network security, access controls |

### Data Flow Privacy Analysis

```
User ──[question]──> Application ──[question + schema]──> HuggingFace LLM API
                          │                                       │
                          │                               [SQL query returned]
                          │                                       │
                          ▼                                       ▼
                     Database ──[query results]──> Application ──> User
```

**Privacy concerns at each step:**

1. **User question → Application**
   - Questions may contain business-sensitive information
   - Questions are logged (log retention policy needed)

2. **Application → HuggingFace API**
   - Database schema is sent to a third-party API (reveals your data model)
   - User questions are sent to a third-party API
   - Query results may be sent for formatting (if using two-call pattern)
   - **This is the biggest privacy risk** — review HuggingFace's data processing agreement

3. **Application → Database**
   - Query results may contain PII
   - Results are processed in application memory
   - Results may be logged

4. **Application → User**
   - Ensure users only see data they're authorized to see

### Required Actions

1. **Data Processing Agreement (DPA) with HuggingFace**
   - Required under GDPR if processing EU personal data
   - Review what data HuggingFace retains and how they use it
   - Review their security certifications

2. **Privacy Policy**
   - What data is collected (questions, usage data)
   - How data is processed (sent to LLM API, stored in logs)
   - Data retention periods
   - User rights (access, deletion, portability)
   - Third-party processors (HuggingFace, hosting provider)

3. **Data Minimization**
   - Don't log full query results (may contain PII)
   - Don't send unnecessary data to LLM API
   - Implement data retention limits (auto-delete logs after X days)

4. **Right to Deletion**
   - Users should be able to request deletion of their query history
   - Must be able to delete from logs, session storage, and any caches

5. **Encryption**
   - Data in transit: HTTPS (TLS 1.2+) for all communications
   - Data at rest: Encrypt database, logs, and session storage

6. **Consider Self-Hosted LLM**
   - If data sensitivity is high, self-hosted LLM eliminates the third-party data risk
   - Trade-off: Higher infrastructure cost, more maintenance

## 6. AI-Specific Regulations

### EU AI Act (if serving EU customers)

The EU AI Act classifies AI systems by risk level:

- **Minimal risk:** Chatbots with clear AI disclosure → likely applies to this product
- **Limited risk:** Requires transparency obligations (tell users they're interacting with AI)
- **High risk:** AI systems making significant decisions → could apply if query results are used for financial/legal decisions

**Required Actions:**
1. Clearly disclose that responses are AI-generated
2. Maintain technical documentation of the AI system
3. Implement human oversight capabilities (user reviews SQL before execution)
4. Keep logs of AI system outputs for regulatory review

### Other AI Regulations
- **US Executive Order on AI** — Voluntary compliance for most companies
- **China AI regulations** — If serving Chinese market
- **Canada's AIDA (proposed)** — Monitor for developments

## 7. Database Access Authorization

### Legal Risk
If the chatbot can query ANY table in a connected database, users might access data they shouldn't see:
- Employee salary information
- Customer PII (addresses, emails, phone numbers, SSNs)
- Financial records
- Medical records
- Authentication credentials

### Required Controls

1. **Table-level access control** — Configure which tables the chatbot can access
2. **Column-level masking** — Mask sensitive columns (e.g., show `john***@email.com`)
3. **Row-level security** — If different users should see different rows
4. **Access logging** — Log who queried what for audit purposes
5. **Regular access reviews** — Periodically review what the chatbot can access

## 8. Intellectual Property in Generated SQL

### Who Owns the Generated SQL?

- The SQL is generated by an AI model (likely Qwen, hosted on HuggingFace)
- Under most jurisdictions, AI-generated content is not copyrightable
- The user's question is their input; the SQL is a mechanical transformation
- **Recommendation:** Terms of Service should state that users can freely use the generated SQL

### LLM Provider Terms

Review HuggingFace's Terms of Service regarding:
- Ownership of model outputs
- Right to use outputs commercially
- Any restrictions on output usage
- Whether they train on inputs (opt-out options)
